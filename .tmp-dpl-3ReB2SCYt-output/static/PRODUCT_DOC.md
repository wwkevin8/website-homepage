# 左邻右里（NGN）产品文档

## 1. 文档说明

本文档基于当前仓库代码整理生成，目标是把现有站点的产品定位、核心功能、页面结构、业务流程、角色权限与数据模型沉淀为一份可继续迭代的产品文档。

适用范围：

- 官网与服务介绍页
- 登录与服务中心
- 接送机 / 拼车服务
- 行李寄存服务

## 2. 产品概述

### 2.1 产品名称

- 中文名：左邻右里
- 英文名：Nottingham Good Neighbor
- 缩写：NGN

### 2.2 产品定位

左邻右里是一个面向诺丁汉留学生的本地生活服务平台，当前以“学生租房咨询与订房承接”为主入口，同时延展出接送机 / 拼车、行李寄存、会员权益、服务中心与后台调度能力，形成“留学落地前后关键节点服务”的组合型产品。

### 2.3 核心价值

- 降低留学生在英国落地阶段的信息不对称
- 把订房、接机、寄存、入住衔接等高频需求整合在一个站点中
- 通过本地团队与后台调度，提高服务响应效率与订单可管理性
- 通过登录体系与服务中心，逐步沉淀用户账号、服务记录与复购入口

## 3. 目标用户

### 3.1 C 端用户

- 即将前往英国诺丁汉的中国留学生
- 已在诺丁汉学习、存在换房 / 暑期寄存 / 接送机需求的学生
- 对本地租房、公寓选择、校园通勤、生活适应存在咨询需求的学生

### 3.2 B 端/内部用户

- 左邻右里顾问团队
- 接送机调度与拼车运营人员
- 后台管理员

## 4. 产品目标

### 4.1 当前阶段目标

- 用官网承接学生咨询流量
- 将接送机与寄存需求在线化收集
- 让公开拼车信息可被用户查看，提升拼车撮合效率
- 通过后台统一管理用户、请求、拼车组与管理员账号

### 4.2 业务目标

- 提升订房咨询转化率
- 提升附加服务下单率
- 降低客服重复沟通成本
- 让拼车调度流程从人工分散记录转向结构化管理

## 8. 核心产品流程

### 8.1 官网咨询转化流程

1. 用户进入官网首页。
2. 了解租房、会员权益、本地团队、合作公寓等信息。
3. 根据需求跳转到接送机、寄存或进一步咨询。
4. 若需要统一登录，则通过 Google 登录进入服务中心。

### 8.2 登录流程

1. 用户进入 `login.html`。
2. 通过 Supabase 发起 Google OAuth。
3. 回调页将 Google 授权结果转换为站内自有会话。
4. 站点通过 `ngn_user_session` 维持登录态。
5. 登录后优先返回原目标页，否则进入服务中心。

### 8.3 接送机 / 拼车流程

1. 用户在 `pickup.html` 了解服务规则、价格与拼车说明。
2. 用户填写 `pickup-form.html`。
3. 系统将请求写入 `transport_requests`，默认状态为 `open`。
4. 后台人员查看请求并人工分组。
5. 已发布且可公开的拼车组展示在公开公告板。
6. 用户通过客服微信继续确认与线下沟通。

### 8.4 寄存服务流程

1. 用户在 `storage.html` 查看非会员价格、规则与估算逻辑。
2. 用户提交寄存预约。
3. 后端写入 `storage_orders`。
4. 系统调用通知能力，向客服或内部渠道发送提醒。
5. 客服后续通过微信/电话人工确认。

### 8.5 后台调度流程

1. 管理员通过后台登录页进入系统。
2. 查看请求列表并按服务类型、机场、状态、日期筛选。
3. 创建或维护拼车组。
4. 将请求加入拼车组，控制可见状态与余量。
5. 对外开放的组通过公共接口展示到前台。

## 9. 功能模块说明

### 9.1 官网首页

目标：

- 传达品牌与服务范围
- 强化“本地团队 + 留学生落地服务”的信任感
- 为租房咨询与附加服务导流

核心内容：

- Hero 品牌主张
- 会员权益卖点
- 为什么选择我们
- 合作公寓/合作方背书
- 校区与生活场景内容
- 团队展示
- 咨询入口

### 9.2 会员权益模块

核心定位：

- 与订房行为绑定的权益说明页
- 作为订房转化的辅助销售内容

当前权益方向：

- 免费接机
- 免费寄存
- 免费搬家
- 新生礼包

### 9.3 团队与内容模块

核心作用：

- 通过真实成员介绍建立信任
- 用“谁擅长解决什么问题”帮助用户降低沟通门槛
- 支持租房咨询转化

### 9.4 登录与服务中心

定位：

- 前台统一账号入口
- 登录后的二次分发页面

设计原则：

- 登录链路尽量轻量
- 手机号非强制，留到真实服务请求阶段再收集
- Google 是当前主登录入口
- 邮箱仍是站内业务主标识

服务中心当前承载：

- 用户头像/昵称/邮箱展示
- 跳转接送机
- 跳转寄存
- 查看会员权益
- 预留后续资料完善入口

### 9.5 接送机 / 拼车模块

用户价值：

- 在留学生高峰接机场景下，降低单人成本
- 提供清晰的服务说明、价格表与操作步骤
- 通过公开公告板提升拼车成功率

前台能力：

- 查看接送机说明
- 查看价格表
- 查看公开拼车信息
- 在线提交接送机/送机请求

后台能力：

- 管理请求
- 管理拼车组
- 控制前台展示
- 管理每组人数与状态

### 9.6 寄存模块

用户价值：

- 面向暑期回国、换房、短期寄放场景提供本地仓储服务

当前能力：

- 非会员价格说明
- 箱型与规格说明
- 在线费用估算
- 在线预约提交
- 预约通知客服

### 9.7 后台用户管理

核心能力：

- 查看站内用户
- 按邮箱或昵称搜索
- 按登录方式筛选
- 查看首次登录、最近登录、登录次数

业务价值：

- 形成基础用户资产沉淀
- 支撑管理员识别用户活跃度

## 10. 接送机业务规则

基于当前代码可确认的规则：

- `service_type` 仅支持 `pickup`、`dropoff`
- 请求状态包括：
  - `draft`
  - `open`
  - `grouped`
  - `closed`
  - `cancelled`
- 拼车组状态包括：
  - `draft`
  - `open`
  - `full`
  - `closed`
  - `cancelled`
- 请求提交时由公共接口写入，并强制设为 `open`
- 拼车组是否前台可见由 `visible_on_frontend` 控制
- 前台公开列表只展示：
  - `visible_on_frontend = true`
  - `status = open`
- 公开列表支持分页、日期范围、机场、服务类型筛选

## 13. 非功能与技术实现特征

### 13.1 技术形态

- 前端为静态多页面站点
- 后端为 Vercel 风格 Serverless API
- 数据库使用 Supabase
- 登录依赖 Supabase + Google OAuth
- 通知使用 `nodemailer` 与内部通知逻辑

### 13.2 安全与权限

- 管理员通过邮箱白名单控制
- 后台页面需要管理员身份
- 非管理员访问后台应返回 `401` 或 `403`
- 接口内包含基础限流能力

### 13.3 国际化

- 前台存在中英双语切换能力
- 站点主要面向中文用户，但保留英文界面支持

## 14. 当前产品优势

- 聚焦留学生高频落地场景，目标人群明确
- 不仅有展示页，也已具备可用的业务提交与后台管理能力
- 已形成“官网承接 + 表单收集 + 后台调度 + 公开公告板”的闭环
- 登录体系与用户表已搭好，为后续会员化和订单中心留出空间

## 15. 当前问题与产品风险

### 15.1 账号与服务记录尚未完全闭环

- 登录用户进入服务中心后，尚不能查看自己的历史请求与订单
- 服务中心目前更像导航页，用户价值仍偏弱

### 15.2 部分流程仍依赖人工客服

- 拼车表单提交后仍需用户手动添加客服微信
- 寄存通知后续也以人工确认为主
- 自动化程度有限

### 15.3 产品边界较宽但优先级尚未完全聚焦

- 订房咨询、接送机、寄存、会员权益、内容展示同时存在
- 若没有清晰的数据指标，资源容易分散

### 15.4 前端编码与内容一致性问题

- 多个 HTML 文件存在中文乱码表现
- 对 SEO、维护体验和后续内容运营不利

## 16. 建议的后续迭代方向

### 16.1 第一优先级

- 增加“我的请求 / 我的订单”页面
- 在服务中心展示用户最近的接送机与寄存记录
- 打通用户与服务记录的关联展示

### 16.2 第二优先级

- 优化拼车流程，降低表单提交后仍需人工二次沟通的成本
- 增加提交成功后的站内状态追踪
- 支持后台更直观地查看待分组请求与剩余座位

### 16.3 第三优先级

- 清理编码问题与文案一致性问题
- 补齐数据埋点与关键转化漏斗
- 评估是否需要支付、优惠券、会员体系深化

## 17. 建议的 KPI

- 官网到咨询点击率
- 接送机页到表单提交转化率
- 寄存页到预约提交转化率
- 拼车请求成组率
- 登录转化率
- 服务中心二次访问率
- 复购用户占比

## 18. 结论

从当前代码看，左邻右里已经不是单纯宣传站，而是一个围绕诺丁汉留学生落地生活服务搭建的轻业务平台。它的核心增长入口仍然是“订房咨询与信任建立”，而真正形成结构化能力的部分，是接送机 / 拼车与寄存的在线化管理。

如果后续继续推进，最值得优先强化的不是再加更多介绍页面，而是把“登录用户 - 服务记录 - 进度反馈 - 后台处理”这条主链路补齐，让平台从“内容+表单站”进一步升级为“可持续运营的服务平台”。
